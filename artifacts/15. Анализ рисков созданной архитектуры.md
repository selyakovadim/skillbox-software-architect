# Анализ рисков созданной архитектуры, компромиссов

### Базовые риски ###
#### Привышение бюджета и сроков проекта ####
__Вероятность наступления__ - средняя  
__Методики невелирования риска:__ 
- найм/хантинг специалистов (обязательно со схожим опытом) высокой квалификации до начала 0 цикла разработки проекта
- работа по гибким/инкрементальным методологиям разработки 
- закупка только простых и доступных всегда элементов инфраструктуры
- минимальное использование "прорывных/модных" решений при разработке кода и выстраивании инфраструктуры 

__Компромиссы:__
- поиск и подключение компании вендора для разработки (необходима дополнительная оценка стоимости)
- использование сторонних шаблонов/фреимворков для разработки приложений (скорость разработки vs управляемость решения)

#### Регуляторные риски ####
__Вероятность наступления__ - средняя  
__Методики невелирования риска:__ 
- своевременный юридический анализ базовой функциональности приложения с учетом специфики законодательства тех стран где планируется внедрение
- написание различных политик и форм отказов от ответственности заблаговременно

__Компромиссы:__
- ничего не делать и положиться на юр.департаменты маркетплейсов приложений (принять риск)
- сопровождение приложения юридическими компаниями (дополнительные затраты)

### Утечка/кража данных пользователей ###
__Вероятность наступления__ - средняя  
__Методики невелирования риска:__ 
- внедрение шифрования при передаче клиентских данных (везде)
- доступ к CMS только из подсети организации с ведением журнала аудита событий ИБ
- внедрение многофакторной (2-х и более) методик авторизации

__Компромиссы:__
- вездесущее шифрование замедлит приложение, необходимо искать баланс
- снизится удобство использования системы управления контентом

### Быстродействие системы (решения в целом) ###
__Вероятность наступления__ - средняя  
__Методики невелирования риска:__
- использовать для, критичных к скорости ответа, данных in-memory хранилища 
- максимально, где это возможно, избавиться от передачи по REST (использовать rpc/grps)
- внедрять практики нагрузочного тестирования и профилирования не только в клиенсткие запросы

__Компромиссы:__
- рассогласованность данных, необходим поиск баланса
- некоторые узлы системы ускорить не так просто (например сторонний рекомендательный сервис)
